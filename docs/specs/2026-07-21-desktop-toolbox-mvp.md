## Problem Statement

用户需要一个桌面端的全能工具集合软件，把日常零散的小工具集中到一个离线优先、启动快速、无需在多个网站间切换的工作台中。首版必须避免把“全能”理解成无限堆砌功能，而是先交付稳定的工具发现、统一交互和高频工具能力。

## Solution

提供一个 Windows 桌面应用的 MVP：左侧按类别浏览、顶部搜索工具、主区域运行工具，并保留最近使用和收藏工具。首批内置四类本地工具：文本、编码转换、文件处理和图片处理；每个工具使用统一的输入、运行、结果预览、复制或另存为交互。默认不上传用户文件或文本。

## User Stories

1. 作为日常办公用户，我想按类别浏览工具，以便快速知道软件能做什么。
2. 作为日常办公用户，我想搜索工具名称和关键词，以便不必逐层寻找工具。
3. 作为用户，我想看到最近使用的工具，以便重复操作时快速进入。
4. 作为用户，我想收藏常用工具，以便建立自己的快捷工作台。
5. 作为用户，我想格式化和校验 JSON，以便处理接口数据和配置内容。
6. 作为用户，我想在 Base64、URL 编码和普通文本之间转换，以便完成常见编码操作。
7. 作为用户，我想计算文本或文件的哈希值，以便核验文件完整性。
8. 作为用户，我想批量重命名本地文件并在执行前预览结果，以便避免误操作。
9. 作为用户，我想压缩或转换本地图片，以便减少文件体积或取得所需格式。
10. 作为用户，我想在所有可能改变文件的工具中先预览，再明确选择输出位置，以便保护原始文件。
11. 作为隐私敏感用户，我想知道处理是否仅在本机完成，以便放心使用文件和文本工具。
12. 作为用户，我想复制文本结果或打开导出的文件位置，以便顺畅地继续后续工作。

## Implementation Decisions

- 首版定位为 Windows x64 桌面应用；采用可跨平台扩展的桌面壳与 Web UI 分层，但不承诺 macOS 或 Linux 发布。
- 使用本地优先的架构：工具运行、最近记录、收藏和设置默认保存在本机；除非未来明确增加联网工具，否则不传输用户内容。
- 建立唯一的“工具注册表”作为测试和扩展切面，统一声明工具的标识、类别、搜索关键词、输入约束、执行器、结果类型和文件写入能力。界面导航、搜索和工具页均由该注册表驱动。
- 工具分为纯转换类和文件输出类。纯转换类可即时执行；文件输出类必须先生成预览，并以另存为或新建输出目录为默认策略，不覆盖源文件。
- MVP 工具范围限定为 JSON 格式化与校验、Base64/URL 编解码、文本/文件哈希、批量重命名预览与执行、图片压缩与格式转换。图片处理的准确支持格式和压缩参数在实现前按所选处理引擎确认。
- 不在首版加入账号、云同步、插件市场、在线 API 聚合、脚本执行器或任意命令执行功能。

## Testing Decisions

- 以工具注册表和公开的工具输入/输出契约为最高测试切面；测试外部行为，不依赖组件内部状态或具体 UI 实现。
- 为每个纯转换工具测试有效输入、无效输入和可复制的标准化输出。
- 为每个文件工具使用临时目录测试预览结果、目标冲突处理、输出内容和“源文件不被修改”的保证。
- 对工具搜索、分类、最近使用和收藏进行端到端行为测试；验证用户能从首页完成一次工具运行并取得结果。
- 在发布前进行 Windows 安装包冒烟测试，覆盖首次启动、离线运行、文件选择、文件导出和错误提示。

## Out of Scope

- macOS、Linux、移动端和浏览器版本。
- 登录、会员、付费、云同步、团队协作和远程文件处理。
- 插件市场、第三方工具下载、在线网页抓取、AI 对话和高风险系统维护工具。
- 覆盖原文件、后台批处理队列和跨设备任务同步。

## Further Notes

- 本规格中的核心验证切面是“工具注册表 + 工具输入/输出契约”；它应同时支撑导航、搜索、执行和测试，避免每新增一个工具都复制一套页面逻辑。
- 规格已发布到 GitHub Issue #1，并带有 `ready-for-agent` 标签；用户确认后已完成 MVP 的初始实现与 Windows 调试包验证。
