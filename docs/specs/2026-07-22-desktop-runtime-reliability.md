## Problem Statement

Windows 桌面版当前有三项基础能力不可用：无边框窗口无法拖动；截图识字在读取剪贴板截图时失败；剪贴板历史显示“正在记录”，但复制其他应用中的文本后仍没有记录。

此外，浏览器页签使用的 `favicon.svg` 中，白色主标与右上角方块发生视觉重合，和 OmniKit 的标准标志不一致。

根因已经定位在桌面端能力配置：剪贴板插件的 `default` 权限不包含任何读写权限，而无边框标题栏没有获得启动窗口拖动的权限。OCR 的剪贴板入口因此在 Windows OCR 运行前就失败，现有界面又将错误收敛成了笼统的“文字识别失败”。

## Solution

为主窗口明确授予拖动窗口、读取文本剪贴板、写入文本剪贴板和读取图片剪贴板的最小权限；保留现有无边框标题栏的拖动区域，使搜索框与窗口控制按钮仍保持可交互。

让剪贴板历史在应用运行期间可靠采集来自其他应用的新文本，并保留现有的去重、置顶、暂停与本地保存行为。让截图识字能读取 `Win + Shift + S` 写入的剪贴板图片并进入现有 Windows 本地 OCR 流程；对 Windows OCR 语言包缺失、剪贴板无图片和原生识别失败分别给出可行动的错误提示。

修正浏览器页签图标的白色图形比例与间距，使主标和右上角方块清晰分离，并保持当前的蓝色 OmniKit 品牌色。

重新整理“关于”页，使它成为介绍 OmniKit 的品牌页面：移除“运行方式、内置工具、界面字体、输出策略”四格信息，不再展示实现细节；用更完整的软件定位、核心价值和本地处理说明替代，并保留必要的在线更新入口。

## User Stories

1. 作为 Windows 用户，我希望按住顶部空白区域即可拖动 OmniKit 窗口，以便在桌面上调整工作位置。
2. 作为 Windows 用户，我希望搜索框可以正常输入、最小化和关闭按钮可以正常点击，以便拖动能力不影响已有窗口操作。
3. 作为使用截图识字的用户，我希望按下 `Win + Shift + S` 截图后能够识别剪贴板中的图片，以便快速提取截图文字。
4. 作为使用截图识字的用户，我希望选择本地图片后仍能完成现有的本地 OCR，以便不依赖剪贴板。
5. 作为使用截图识字的用户，我希望在剪贴板没有图片、Windows 缺少 OCR 语言功能或识别失败时看到原因和下一步操作，以便自行解决问题。
6. 作为使用剪贴板历史的用户，我希望复制其他应用中的文本后能在 OmniKit 中看到该记录，以便再次使用。
7. 作为使用剪贴板历史的用户，我希望暂停记录后不再收集新内容，恢复记录后继续收集，以便控制隐私与干扰。
8. 作为使用剪贴板历史的用户，我希望重复复制相同文本时不会产生无意义的连续重复项，以便历史保持可读。
9. 作为隐私敏感用户，我希望文字、图片识别和剪贴板记录均只在本机处理，以便不将内容上传到网络。
10. 作为桌面版用户，我希望应用重启后保留已有的剪贴板历史、置顶和暂停状态，以便使用习惯不中断。
11. 作为查看网页或 GitHub 页面的人，我希望浏览器页签中的 OmniKit 图标清晰、没有图形重叠，以便快速识别产品。
12. 作为打开“关于”页的用户，我希望看到清晰、美观的软件介绍，而不是运行方式和字体等实现信息，以便快速理解 OmniKit 的价值。

## Implementation Decisions

- 主窗口能力配置将显式允许启动窗口拖动，以及剪贴板文本与图片的必要读写操作；不授予未被使用的剪贴板 HTML、清空或图片写入能力。
- 无边框标题栏继续作为唯一的拖动入口。搜索输入、快捷键提示和窗口控制按钮必须保持非拖动交互，避免拖动手势吞掉点击或输入。
- 剪贴板历史继续使用当前的运行期采集与本地持久化模型；读取失败不得被显示成“正在记录”，而应保留最近的有效记录并给出明确的不可用状态。
- 截图 OCR 继续使用 Windows 内置 OCR，不增加云端服务或第三方识别模型。剪贴板图片读取成功后再调用现有原生识别命令；临时图片必须在识别后删除。
- OCR 错误按“未检测到剪贴板图片”“Windows OCR 语言功能不可用”“图片无法解码或没有可识别文字”“其他原生错误”区分呈现，保留用户可以据此操作的中文提示。
- 浏览器 `favicon.svg` 使用独立的白色主标和右上角方块，不允许路径、描边或圆角边界互相覆盖；尺寸缩小后仍应保持可辨识的留白。
- “关于”页移除四格实现信息，不显示运行方式、内置工具数量、界面字体和输出策略。页面主体改为 OmniKit 的产品定位、文本/文件/图片工具的覆盖范围、本地优先与可控输出的价值说明；保留品牌标志、版本号、隐私说明和在线更新入口。
- “关于”页视觉保持现有浅蓝品牌基调，以更清晰的层级、留白和小范围动效优化展示，不增加营销轮播、外部链接墙或部署说明。
- 不调整工具布局、历史容量、OCR 支持语言范围或在线更新流程。

## Testing Decisions

- 在权限配置层增加回归检查，确保主窗口只拥有拖动与当前实际使用的剪贴板读写权限。
- 为剪贴板历史的采集边界补充行为测试：首次读取新文本会新增记录、连续重复文本不会新增、暂停时不新增、恢复后可以继续新增，并继续覆盖现有置顶和容量规则。
- 为 OCR 的前端错误映射补充测试，覆盖无剪贴板图片、原生 OCR 返回语言包提示和成功返回文字三种外部行为。
- 在 Windows 打包后的应用上进行冒烟测试：拖动标题栏空白处、输入搜索、点击窗口控制按钮；复制外部文本后观察历史；使用 `Win + Shift + S` 识别截图；选择一张本地图片识别。
- 检查浏览器页签图标在常见的小尺寸下的渲染，确认主标与右上角方块之间没有视觉重叠。
- 在桌面宽度与窄窗口宽度下检查“关于”页，确认已不显示四格实现信息，介绍内容、版本与更新入口均可读且可操作。
- 验收以实际桌面端行为为准，不以组件内部状态、定时器实现或具体原生 API 调用次数作为测试断言。

## Out of Scope

- 全局后台剪贴板守护、应用关闭后的持续记录、图片历史记录和跨设备同步。
- 接入云端 OCR、离线模型下载、手写识别、表格结构识别或 OCR 结果自动翻译。
- 窗口尺寸策略、整体界面重构、工具功能扩充和在线更新机制调整。
- 增加新工具、宣传轮播、用户账号、社区链接或产品部署说明。
- 对 macOS、Linux 或浏览器版提供相同的系统剪贴板与 Windows OCR 行为。

## Further Notes

- 修复后仍需用户在 Windows 中安装中文或英文 OCR 可选语言功能；缺少该系统组件时应用应明确说明，而不是显示泛化失败提示。
- 验收环境应使用正式 Windows 安装包，因为浏览器预览没有系统窗口和系统剪贴板权限。
